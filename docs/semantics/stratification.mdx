# Stratification

Semi-naive evaluation handles recursion beautifully. But some rules can't just be thrown into the same pot — **negation** and **aggregation** need to know the final answer before they can run. That's where stratification comes in.

## The problem

Consider this program:

```flowlog
Reach(x, y) :- Edge(x, y).
Reach(x, z) :- Reach(x, y), Edge(y, z).

Unreachable(x, y) :- Node(x), Node(y), !Reach(x, y).
```

`Unreachable` negates `Reach`. But if we evaluate all three rules simultaneously, `Reach` is still growing while `Unreachable` tries to read it. Negating a half-computed relation gives wrong answers — you'd "discover" that node 5 is unreachable, only to find out two rounds later that it actually is reachable.

The same issue applies to aggregation. You can't compute `min(cost)` while new, potentially smaller costs are still being derived.

## The solution: layers (strata)

<StyledFlowLog /> splits rules into ordered layers called <b>strata</b>. Each stratum runs to fixpoint before the next one starts. The rule: if relation A <b>negates</b> or <b>aggregates</b> over relation B, then B must be in an earlier stratum.

The compiler figures this out automatically:

1. **Build a dependency graph** — an edge from B to A means "A depends on B."
2. **Find strongly connected components (SCCs)** — relations that mutually depend on each other through *positive* edges go in the same stratum. They can recurse together safely.
3. **Order by negation/aggregation edges** — these edges impose a strict ordering between strata. B's stratum must finish before A's stratum begins.

## Walkthrough

```flowlog
.decl Edge(x: int32, y: int32)
.decl Node(x: int32)
.decl Reach(x: int32, y: int32)
.decl Unreachable(x: int32, y: int32)
.decl CC(node: int32, label: int32)

// Group 1: Reach depends on itself (positive recursion)
Reach(x, y) :- Edge(x, y).
Reach(x, z) :- Reach(x, y), Edge(y, z).

// Group 2: CC also depends on itself (positive recursion)
CC(x, min(x)) :- Node(x).
CC(x, min(c)) :- Edge(x, y), CC(y, c).

// Group 3: Unreachable negates Reach
Unreachable(x, y) :- Node(x), Node(y), !Reach(x, y).
```

The compiler produces three strata:

| Stratum | Relations | Why |
|---------|-----------|-----|
| 1 | `Reach` | Recursive, no negation/aggregation dependencies |
| 2 | `CC` | Uses `min` aggregation (self-contained, no dependency on Reach) |
| 3 | `Unreachable` | Negates `Reach` — must wait for stratum 1 to finish |

Strata 1 and 2 are independent and could even run in parallel. Stratum 3 must wait for stratum 1.

## What happens in each stratum

Each stratum runs [semi-naive evaluation](semi-naive) independently:

1. **Load inputs** — input facts and results from earlier strata.
2. **Semi-naive to fixpoint** — apply rules, track deltas, iterate until Δ = ∅.
3. **Freeze** — the results are now final. Later strata can safely negate or aggregate over them.

## Negation in recursion

What if negation appears inside a recursive cycle?

```flowlog
A(x) :- Node(x), !B(x).
B(x) :- Node(x), !A(x).
```

A needs B to be finished, and B needs A to be finished — a chicken-and-egg problem. There's no valid stratification order.

<StyledFlowLog /> will <b>warn</b> about this but still compile the program. The semantics become your responsibility: the engine evaluates the rules to a fixpoint, but the result may depend on evaluation order and may not match well-founded or stable-model semantics. If you know what you're doing (e.g., you've proven your program converges), go for it — just be aware that the compiler can't guarantee correctness here.

## Aggregation in recursion

Same story for aggregation inside a recursive cycle:

```flowlog
SSSP(x, min(0)) :- Source(x).
SSSP(y, min(d + w)) :- SSSP(x, d), Edge(x, y, w).
```

Here `SSSP` aggregates over itself — `min` appears in the head of a recursive rule. Strictly speaking, this breaks stratification (the aggregate reads a relation that's still being computed).

<StyledFlowLog /> allows this with a <b>warning</b>. In practice, monotone aggregates like `min` and `max` converge to the correct fixpoint because the aggregate value can only move in one direction. But the compiler doesn't verify monotonicity — it trusts you. If you use a non-monotone aggregate (like `count` or `average`) recursively, the result may not be meaningful.

## When it's safe vs when to be careful

| Pattern | Safe? | Notes |
|---------|-------|-------|
| Negation across strata | Yes | Fully stratified, compiler handles it |
| Aggregation across strata | Yes | Same — earlier stratum finishes first |
| `min`/`max` in recursion | Usually | Monotone, converges in practice — but you own the proof |
| `count`/`sum`/`average` in recursion | Risky | Non-monotone — fixpoint may not be meaningful |
| Negation in recursive cycle | Risky | No valid stratification — result is evaluation-order dependent |
