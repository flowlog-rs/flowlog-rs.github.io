# Data Types

<StyledFlowLog /> supports the following primitive data types for relation columns:

## Integer

| Type   | Size    | Min | Max |
|--------|---------|-----|-----|
| `i8`   | 8-bit   | -128 | 127 |
| `i16`  | 16-bit  | -32,768 | 32,767 |
| `i32`  | 32-bit  | -2.1 × 10⁹ | 2.1 × 10⁹ |
| `i64`  | 64-bit  | -9.2 × 10¹⁸ | 9.2 × 10¹⁸ |

Integer types map directly to the corresponding Rust signed integer types in the generated code.

```flowlog
.decl Score(student_id: i32, grade: i8)
// Score(1001, 95).
```

## String

The `string` type represents a UTF-8 string value. String constants are written in double quotes:

```flowlog
.decl Name(id: i32, label: string)
// Name(1, "alice").
```

## Boolean

The `boolean` type represents a true/false value:

```flowlog
.decl Flag(id: i32, active: boolean)
// Flag(1, true).
```

## Declaring types in relations

Types are specified in `.decl` directives:

```flowlog
.decl Edge(src: i32, dst: i32)
.decl Person(id: i64, name: string, active: boolean)
.decl Score(student: i16, grade: i8)
```

Every column in a relation must have an explicit type annotation.

## Type checking

<p><StyledFlowLog /> enforces type safety at relation boundaries — <b>Joins</b>, <b>Rule heads</b>, and <b>String/Boolean constants</b>. Integer constant overflow and comparison operand types are not checked by <StyledFlowLog /> and will be caught by the downstream Rust compiler.</p>

### Joins

Variables used as join keys across relations must have matching types. For example, joining `Edge(x, y)` with `Node(y, label)` requires `y` to have the same type in both `Edge` and `Node`.

```flowlog
.decl Edge(src: i32, dst: i32)
.decl Node(id: i32, label: string)

// Valid: y is i32 in both Edge and Node
Match(x, label) :- Edge(x, y), Node(y, label).
```

<p>If <code>Node</code> were instead declared as <code>Node(id: i64, label: string)</code>, <StyledFlowLog /> would report a type error because <code>y</code> cannot be both <code>i32</code> and <code>i64</code>.</p>

### Rule heads

When multiple rules write to the same IDB relation, all rules must produce the same column types as the IDB declaration. For example, if `Route(src, dst, cost)` is declared with `(i32, i32, i16)`, every rule deriving `Route` must produce `(i32, i32, i16)`.

```flowlog
.decl Edge(src: i32, dst: i32)
.decl Weight(src: i32, dst: i32, cost: i16)
.decl Route(src: i32, dst: i32, cost: i16)

// Both rules produce (i32, i32, i16) — matches the declaration
Route(x, y, c) :- Edge(x, y), Weight(x, y, c).
Route(x, z, c) :- Route(x, y, _), Edge(y, z), Weight(y, z, c).
```

Since `x`, `y`, and `z` come from `Edge` (declared as `(i32, i32)`) and `c` comes from `Weight` (declared with `cost: i16`), the types match the `Route` declaration.

### Constants

<p><code>String</code> and <code>Boolean</code> constants are type-checked by <StyledFlowLog />. Integer constants (e.g., <code>42</code>) are <b>NOT</b> explicitly checked — if a constant doesn't fit the column type (e.g., <code>999</code> in an <code>i8</code> column), the value will be silently truncated.</p>

### Comparisons

<p>Type compatibility of comparison operands (e.g., <code>x &gt; y</code>, <code>x != 3</code>) is not checked by <StyledFlowLog /> directly. Mismatched types will produce a compilation error in the generated code.</p>
